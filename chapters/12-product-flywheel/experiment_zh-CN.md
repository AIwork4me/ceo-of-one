# 第 12 章：产品飞轮 — 构建自我修复的产品

---

## 背景

第 1-11 章教你用 AI 构建产品。你现在有一个 Next.js 应用，111 个测试、CI、Vercel 部署。现在加最后一块：一个在你睡觉时自动修复 bug、发布改进的系统。

## 任务

为你的项目搭建产品飞轮：
1. 自动监控 GitHub issues
2. 启动 AI 代理修复
3. 合并前运行 CI
4. 部署到线上
5. 关闭 issue 并通知你

## COO 流程

### 1. 回顾

之前章节的关键教训：

- **测试先行**（第 3 章）：验收标准防止 bug；测试抓住回归
- **架构先于功能**（第 4 章）：模块化结构让定向修复更安全
- **CI 作为门禁**（第 8 章）：测试和构建不通过不合并
- **单任务陷阱**（本章）：修复和合并必须是独立进程

### 2. 审计

检查你已有的东西：

```
# 1. CI 工作流存在吗？
ls .github/workflows/ci.yml

# 2. 测试通过吗？
cd platform && npx jest --ci --forceExit

# 3. master 分支有保护吗？
gh api repos/{owner}/{repo}/branches/master/protection
```

**我们的配置：** GitHub Actions CI，111 个测试 + 构建。没有分支保护——我们用合并器作为门禁，只在 CI 通过时合并。

### 3. 架构决策

三种架构对比：

| 架构 | 延迟 | 复杂度 | 失败模式 |
|------|------|--------|---------|
| 单个定时任务 | 低 | **不可行** — 修复代理还没完成，合并步骤就跑了 | 静默失败：报告"没有 issue"，实际修复正在进行 |
| GitHub Actions bot | 中 | 高 — CI 里需要 API key，无本地环境 | 密钥泄露风险 |
| **两个定时任务** | 低 | 中（两个配置） | 干净分离 — 每个任务一个职责 |

**决策：两个定时任务。** 修复器（每 4 小时）和合并器（每 30 分钟）是独立进程，有独立的失败模式。一个坏了，另一个继续工作。

### 4. 细化

**修复器**（每 4 小时运行）：
- 从 GitHub API 拉取开放 issue
- 跳过带 `needs-review` 标签的
- 跳过已有 PR 或进行中分支的
- 对每个合格 issue：在隔离分支上启动 Claude Code 代理
- 代理读取 issue、修复、跑测试、创建带 `auto-fix` 标签的 PR
- PR body 包含 `Closes #{issue_number}` 实现自动关闭
- Fire-and-forget：不等代理完成

**合并器**（每 30 分钟运行）：
- 找到带 `auto-fix` 标签的开放 PR
- 检查 `mergeable_state`：只在 `clean` 时合并
- `unstable`：检查 CI 状态，运行中则等待，失败则中止
- `clean`：squash 合并，从 PR body 提取 issue #，关闭 issue
- `blocked` 或 `behind`：报告，不合并
- **始终报告状态** — 即使无事可做

**安全规则：**
- 从不强制推送
- CI 失败不合并
- 失败时标记 `needs-review` → 通知人工
- Claims 文件防止重复处理
- 每轮最多 10 个 issue
- 每次修复最多 20 轮 Claude Code

### 5. 执行

取决于你的平台。OpenClaw 配置：

创建两个定时任务：
- **Issue Fixer**：每 4 小时，拉取 issue → 启动修复代理
- **Merge Watcher**：每 30 分钟，检查 PR → CI → 合并 → 关闭 issue

**CI 工作流** (`.github/workflows/ci.yml`)：
- PR 和 push 到 master 触发
- 跳过纯文档变更（节省资源）
- 111 个测试 + 构建

**GitHub 标签**：
- `auto-fix` — 绿色，待合并 PR
- `needs-review` — 红色，需人工
- `auto-fixed` — 蓝色，归档

### 6. 验证

**端到端测试：**

1. 在仓库创建一个真实 issue，带清晰验收标准
2. 等修复器触发（或手动触发）
3. 观察：PR 应在 5 分钟内出现，带 `auto-fix` 标签
4. 观察：CI 自动运行
5. 观察：合并器在 CI 通过后 30 分钟内合并
6. 验证：Issue 自动关闭
7. 验证：生产部署上线

**我们的真实数据：**

| Issue | 描述 | 修复时间 | 上线总耗时 |
|-------|------|---------|-----------|
| #17 | Hero 重设计（before/after 叙事） | 1m52s | ~3.5 分钟 |
| #19 | OG 图片重设计（视觉叙事） | 3m26s | ~5 分钟 |

### 7. 记住

- B @双任务架构：修复和合并必须是不同频率的独立定时任务。证据：单任务方案在实践中失败。
- B @CI-门禁：CI 是坏修复和生产之间唯一的安全网。证据：每个 PR 跑 111 测试 + 构建。
- B @标签状态机：三个标签处理所有状态。证据：10+ 真实 issue 的干净状态转换。
- B @fire-and-forget：子代理在隔离分支运行，失败不影响 master。证据：尽管有代理失败，生产环境没有错误合并。
- B @始终报告：每次定时运行必须报告状态。证据：静默失败是最严重的失败。
- O(c=0.90) @反馈延迟：用户反馈 → 修复部署时间是最重要但大多数团队不测量的产品指标。我们：3.5-5 分钟。
- O(c=0.85) @运营思维：从"写代码"到"运营系统"的转变是本课程最重要的思维转变。

### 8. 报告

**结果：**
- 产品飞轮运行中：✅
- 两个定时任务：修复器（4h）+ 合并器（30min）
- CI 门禁：111 测试 + 构建，必须通过才合并
- 标签体系：auto-fix / needs-review / auto-fixed
- 端到端验证通过：Issue → PR → CI → 合并 → 部署 → 关闭
- 真实延迟：3.5-5 分钟从 issue 到生产

**这对你的产品意味着什么：**
你的用户将体验到一种罕见的东西：一个在几分钟内回应他们反馈的产品。不是因为你更努力——而是因为你构建了正确的系统。飞轮自动将用户反馈转化为产品改进。它转得越快，你得到越多反馈，越多改进上线，用户越信任你。

这是第 12 章。你现在拥有了一切：产品、团队（AI 团队）、质量体系和自我改进闭环。还剩最后一章。
