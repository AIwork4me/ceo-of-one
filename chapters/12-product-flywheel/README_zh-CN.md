# 🔄 产品飞轮

---

## 🎯 你将学到

- 为什么**发布是开始，不是结束** — 真正的产品从用户反馈中进化
- 如何搭建**全自动修复管线**：issue → 修复 → 测试 → 部署 → 关闭
- 为什么**两个独立的定时任务**（而不是一个）是唯一可行的架构
- 为什么**CI 是最后一道防线** — 跳过它可能是你做过的最昂贵的决定

大多数教程写到"部署"就结束了。你发布了产品，庆祝一下，然后……什么都没有。产品就停在那里。用户发现 bug，提交 issue，然后等。有时候等几天，有时候永远等不到。

这一章讲的是发布之后的事。因为一个不对用户反馈做出响应的产品就是死产品——不管代码写得多好。

## 活的产品

想想你欣赏的产品。不是那些一发布就完美的——而是那些**持续变好**的。它们有一个共同点：反馈闭环短于一天。

用户报告问题。等他们再来看的时候，已经修好了。不是因为有人在旁边盯着——而是系统被设计成能自我修复。

这不是科幻。我们做到了。以下是原理。

### 死产品模式

大多数独立开发者遵循这样的循环：

1. 做产品 → 部署 → 庆祝
2. 用户发现 bug → 提交 issue
3. 开发者查看 issue（明天？下周？）
4. 开发者修 bug → 测试 → PR → 合并 → 部署
5. 回到第 2 步

第 2 步到第 4 步之间的间隔，就是产品死亡的地方。一个 issue 每多放一天，用户就多失去一分信任。当间隔变成一周，用户就不再报告 bug 了。他们直接离开。

### 活的产品模式

飞轮把间隔压缩到接近零：

1. 用户提交 issue
2. **系统**检测到（4 小时内）
3. **系统**启动 AI 代理修复（约 2-5 分钟完成）
4. **系统**创建带测试的 PR
5. **系统**运行 CI（测试 + 构建）
6. **系统**在 CI 通过后合并
7. **系统**部署到线上
8. **系统**关闭 issue 并通知你

从用户视角看：他们报告了一个问题，回来查看时已经修好了。他们没有等你。产品自己修好了自己。

这就是**发布**产品和**运营**产品的区别。

                               ┌─────────────┐
                               │ Claude Code  │
                               │ 读取 issue   │
                               │ 修复代码     │
                               │ 编写测试     │

                               │ (auto-fix)   │
                               └──────┬──────┘
                                      │
                                      ▼

                               CI: 111 个测试
                               + 构建
                                      │

                   │  Merge Watcher   │◄┘  每 30 分钟
                   │  (定时任务 2)    │
                   │  检查 CI        │

                   └───────┬─────────┘

                           │

六个环节，一个循环。

## 两个任务，一个循环

我们犯的第一个错误：把所有东西放在一个定时任务里。流程是：

```
拉取 issue → 启动修复代理 → 等待完成 → 检查 CI → 合并 → 通知
```

纸上看起来没问题。实际上立刻就崩了。

问题出在：`启动修复代理`是 fire-and-forget（发完就不管了）。代理在独立进程里跑，需要 2-5 分钟。但定时任务不会等——它直接跑到"检查 CI"，发现什么都没有（PR 还没创建），然后报告"没有发现 issue"。修复代理 3 分钟后完成，但没人听到。

**解决方案：拆成两个任务。**

| 任务 | 频率 | 做什么 | 为什么分开 |
|------|------|--------|-----------|
| **Issue Fixer** | 每 4 小时 | 检测 issue → 启动代理 → 创建 PR | 修复慢（2-5 分钟）且不可预测 |
| **Merge Watcher** | 每 30 分钟 | 检查 PR → 验证 CI → 合并 → 关闭 issue | 合并快（瞬时完成）但需要 CI 跑完 |

修复器低频运行，因为用户不会每分钟提交 issue。合并器高频运行，因为 PR 一旦创建，你希望 CI 通过后立刻合并——而不是 4 小时后。

**最坏情况：** issue 在修复器运行 3.5 小时后提交 + 5 分钟修复 + 30 分钟等合并器 = **约 4 小时上线。**

**最好情况：** issue 正好在修复器运行时提交 + 5 分钟修复 + 30 分钟等合并器 = **约 35 分钟上线。**

真实数据：
- Issue #17（Hero 重设计）：1 分 52 秒创建 PR，总计约 3.5 分钟上线
- Issue #19（OG 图片）：3 分 26 秒创建 PR，CI 通过后合并

这种职责分离不是可选项，是唯一可行的架构。

## CI：安全网

每个 PR 在合并前都必须经过 GitHub Actions：

- 运行 111 个测试
- 执行生产构建

**不可跳过。** 原因：

AI 代理很聪明，但不是不会犯错。它偶尔会引入回归——改了一个组件导致另一个组件坏掉，加了一个依赖和现有依赖冲突，或者"修好"了一个 bug 却破坏了别的东西。CI 能抓住这些。

**CI 还会跳过纯文档变更**，节省资源。Markdown 文件不需要 111 个测试来验证。

**CI 失败了怎么办？** 合并器看到失败状态后停止。**不合并。** 通知你失败详情。你审查后手动修复，或等下一轮重试。

**永远不要跳过 CI。** 跑 90 秒的测试是你买过的最便宜的保险。一次错误合并可能要花几个小时调试和失去用户信任。

## 标签即状态机

标签是飞轮追踪处理状态的方式：

| 标签 | 含义 | 动作 |
|------|------|------|
| `auto-fix` | 飞轮创建的 PR | 合并器自动接手 |
| `needs-review` | 修复失败或需人工判断 | 通知人工，跳过自动合并 |
| `auto-fixed` | issue 已解决并部署 | 归档标记 |

流程很简单：

```
Issue 创建（无标签）
  → 修复代理创建 PR，标记 "auto-fix"
  → 合并器发现 "auto-fix" → 检查 CI → 合并
  → Issue 自动关闭（PR body 含 "Closes #XX"）
```

**任何环节出错**，issue 会被标记 `needs-review` 并通知你。飞轮从不强制合并，从不强制推送。它安全地失败。

## 这意味着什么

坦白说，你在这 12 章里做了什么。

你从零开始。CEO 说了一句话。AI COO 把它拆解、细化，交给 AI 工程师去构建。12 章、12 个模块、111 个测试、一个双语网站，而现在——一个在你睡觉时还在持续改进产品的系统。

这不再是教程了。这是一个生意。

**哲学转变：** 大多数开发者认为自己的工作是写代码。COO 认为自己的工作是运营系统。代码只是其中一个组件。CI 管线、标签、定时任务、通知渠道——这些都是组件。像任何系统一样，每个组件有角色，系统只在角色清晰、交接干净时才工作。

**实际意义：** 你现在拥有一个在几分钟内响应用户反馈的产品。这不是功能——这是竞争优势。用户会注意到他们的反馈消失了又以修复的形式出现。他们会告诉朋友。他们会提交更多反馈。飞轮转得更快。

**下一步：** 第 13 章是你的毕业项目。用学到的所有知识，做一个完全属于你自己的东西——从想法到收入，配上完整的产品飞轮。
