# 第五章：认证 — 在干净架构上添加第一个功能

## 背景

第四章搭完落地页后，应用需要真正的功能。认证是最显而易见的第一步：登录、注册、令牌管理。架构很干净——还没有任何 feature，只有第三章留下的骨架。这应该是简单的一章。

确实简单。简单到危险。

## 任务

构建完整的认证模块：类型定义、状态管理、API 服务、路由守卫和测试。CEO 说了一句"继续阶段七"，COO 组装了一份详细提示词发给 Claude Code。

## 实际发生了什么

- Claude Code 构建：types、store、service、routes、31 个测试
- **1 个 TypeScript bug**：`result.errors` 在未做类型收窄的情况下访问了判别联合体 —— 被 Claude Code 自动修复
- 65/65 测试通过，构建干净
- 总耗时：很快

## 哪里出了问题

### 1. COO 事后没有做架构审计

没有结构性审查。没有用 `grep` 搜索跨 feature 导入。没有检查依赖图。COO 验证了"能不能用"（测试通过），但从未验证"架构是否干净"。到了第六章，这种结构性检查的缺失会狠狠地反噬。

### 2. Bug 是规范失败，不是代码失败

TypeScript 类型收窄 bug 的发生，是因为 COO 的提示词没有说明如何处理判别联合体类型。一句"对判别联合体使用显式类型守卫"就能预防。这个 bug 不是随机的——它是提示词不够完整造成的直接后果。

### 3. "自愈代码"被错误地包装成了优点

原始实验记录把自动修复包装成了亮点："代码自愈了！"但这不是优点。这是 COO 规范的缺陷，只是碰巧被 Claude Code 捕获了。把它说成自愈，掩盖了真正的教训：**COO 的提示词差了一条指令才完整。**

### 4. 实验记录是事后补写的

子代理根据上下文写了原始记录，而不是根据真实体验。那是文档，不是学习。记录看起来很漂亮，但没有任何真正的洞见——因为没有进行过真正的反思。

### 5. COO 在充当提示词写手，而不是决策者

COO 组装提示词、发送、检查结果。那是提示词写作。真正的 COO 应该做架构决策、定义约束、验证结构完整性。角色已经在退化——只是因为任务简单所以还不明显。

## 真正学到的东西

| 类型 | ID | 洞见 |
|------|----|------|
| **B** | @Ch5-bug | TypeScript 类型收窄 bug 由不完整的规范导致。如果 COO 指定了"对判别联合体使用显式类型守卫"，bug 就不会存在。每一个被自动修复的 bug 都是 COO 未能填补的规范缺陷。 |
| **O**(0.85) | @first-feature | 在干净架构上添加第一个功能具有欺骗性的容易。真正的考验是第二和第三个功能。Ch5 有 1 个 bug，Ch6 有 3 个，Ch7 有 0 个——模式不是线性的。 |
| **B** | @verification-gap | 65 个通过的测试掩盖了结构性检查的缺失。COO 验证了"能不能用"，但没验证"架构是否干净"。功能测试是必要的，但不够。 |

## 欺骗性的一面

第五章感觉像是成功了。65/65 测试通过，构建干净，一个小 bug 自动修复了。COO 报告"一切正常"，然后继续前进。

问题在于：COO 什么都没学到。Bug 被忽略了，结构性审计被跳过了，角色继续退化成"快速提示词组装器"。第六章会暴露这一切。

## COO 复盘评分

- **规范质量**：7/10 — 基本完整，但遗漏了类型收窄
- **结构性验证**：0/10 — 完全没有执行
- **自我认知**：3/10 — 没有识别到规范缺陷或角色退化
- **学习提取**：2/10 — 记录了发生了什么，而不是意味着什么
