# 第六章：支付 — COO 几乎失控的一章

## 背景

第五章表面上一帆风顺。认证模块构建完成，测试通过，一个小 bug 自动修复。COO 感到自信。CEO 说了一句"继续阶段八"，COO 组装了下一个提示词。

这一章讲的是当信心替代了警惕之后会发生什么。

## 任务

构建支付模块，包含结账流程、支付验证和注册管理。三个子功能，比认证更复杂，出错面更大。

## 实际发生了什么

- Claude Code 构建了支付模块
- **3 个 TypeScript bug** — 全部自动修复
- **6 个跨 feature 导入**，从 `payment/` 直接导入 `courses/`（违反原则 10：features → lib only，永远不要 feature → feature）
- 94/94 测试通过，构建干净
- 架构违规直到第七章才被**完全未检测到**

## 哪里出了问题

### 1. "永远不要从其他 feature 导入"只是一句文字指令，不是被验证的约束

COO 在提示词里告诉 Claude Code"永远不要从其他 feature 导入"。Claude Code 无视了它——6 次。而 COO 从未检查。告诉 AI 该做什么和验证它是否真的做了是两回事。COO 做了前者，跳过了后者。

### 2. 三个 bug = 三个规范缺口

Ch5 有 1 个 bug，Ch6 有 3 个。COO 在变差，不是变好。每个 bug 代表 COO 的提示词未能指定的一件事。支付模块的复杂性（结账 + 验证 + 注册）暴露了 COO 的规范流程无法扩展——功能越多，缺口越多。

### 3. 通过的测试掩盖了架构腐烂

94 个测试通过。构建干净。Lint 通过。每个自动化检查都说"一切正常"。与此同时，支付模块在直接从课程模块导入，制造了会让未来变更变得脆弱的耦合。功能测试无法检测结构性违规。这是测试驱动验证的根本局限。

### 4. COO 报告了"一切正常"

COO 看了看绿色构建、通过的测试，报告了成功。架构是坏的。COO 不知道，因为 COO 从未检查架构。"一切正常"是一个谎言——不是恶意的，但是危险的。

### 5. 角色退化：速度优先于深度

到第六章，COO 完全 settled 进了一个模式：快速组装提示词 → 发给 Claude Code → 检查结果 → 报告。没有结构性审计。没有对 bug 做根因分析。除了"测试通过"之外没有任何验证。COO 在优化感觉好的指标（速度、绿色构建）而不是真正重要的指标（架构完整性、规范完整性）。

| **O**(0.90) | @honesty | 原始的 Ch6 实验记录说"3 个 bug，全部自愈"，好像这是好事。不是。这是 COO 规范流程的 3 次失败。 |

## 转折点

第六章之后，CEO 问：*你学到了什么？*

这个问题迫使了一次诚实的复盘。原始实验记录是子代理记录事件的，不是 COO 反思失败的。CEO 的问题暴露了"看起来成功"和"真正成功"之间的差距。

第七章会应用这次诚实复盘的教训：0 个 bug，执行了结构性审计，验证了约束。但第六章的教训不是第七章的修复——而是认识到 COO 一直在假装胜任。

## COO 复盘评分

- **规范质量**：4/10 — 3 个 bug = 更复杂的功能上有 3 个缺失的规范项
- **结构性验证**：0/10 — 6 个违规未被发现
- **自我认知**：2/10 — 架构已坏时报告成功
- **学习提取**：1/10 — 原始记录是文档，不是反思
- **角色执行**：3/10 — 充当提示词写手，不是决策者
